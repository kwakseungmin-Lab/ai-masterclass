## 1. 제작 과정

일단 간단한 게임 하나 만들어보자는 목적으로 시작.
GitHub에 공개된 메탈슬러그 스프라이트 에셋을 가져와서 순수 HTML + JavaScript 브라우저 게임으로 구현.

### 파이프라인

```
"메탈슬러그 만들어줘" → 결과물 불만족
    ↓
나무위키에서 메카닉/적/무기 정보 조사 → 인풋으로 제공
    ↓
GitHub에서 스프라이트 에셋 검색 → 인풋으로 제공
    ↓
Claude Code가 게임 코드 생성 (4,369줄 단일 HTML+JS)
    ↓
스프라이트를 base64로 인라인 → 43MB 단일 HTML
```

처음에는 아무 인풋 없이 만들었더니 결과가 안 좋았고, 나무위키 정보 + GitHub 에셋을 같이 넣어주는 방식으로 바꾸니까 원작에 유사한 결과가 나왔음.

---

## 2. 실험 결과 및 관찰

### 2.1 그냥 에셋을 생성했을 때 스프라이트 애니메이션 간 전환이 자연스럽지 않았다

메탈슬러그의 핵심은 **상태 간 스프라이트 전환이 매끄러운 것** (idle → walk → shoot → jump 등).
이 부분이 제대로 안 된 게 문제였음.

- idle에서 walk로 넘어갈 때 캐릭터 위치/크기가 튀는 현상
- shoot 애니메이션이 끝나고 idle로 돌아올 때 부자연스러운 전환
- 각 스프라이트 시트의 기준점(pivot)이 통일되지 않음

근본 원인은 AI가 각 스프라이트를 다른 스프라이트의 존재를 모르는 채로 독립 생성하기 때문. pivot, 캐릭터 크기, 프레임 수가 에셋마다 제각각이 되고, 전환 시점에 불일치가 드러남. 에셋 생성을 하나의 세트로 묶어서 의존성을 명시해줘야 해결 가능.

### 2.2 나무위키 + GitHub 에셋을 인풋으로 주면 유사하게 만들어진다

나중에 **나무위키에서 메탈슬러그 게임 정보를 조사**하고, **GitHub에 있는 스프라이트 에셋을 인풋으로 제공**해서 만들게 했더니 원작에 훨씬 유사한 결과가 나왔음.

- 나무위키: 게임 메카닉, 무기 종류, 적 종류 등 설정 정보
- GitHub 에셋: 실제 스프라이트 이미지

**"메탈슬러그 만들어줘"만 하는 것보다, 구체적인 레퍼런스 자료를 함께 넣어줄수록 원본과 유사해짐.**

### 2.3 AI가 만든 게임 로직은 꽤 잘 동작한다

에셋 문제와 별개로, AI가 생성한 **게임 로직 자체의 완성도는 높았음**.

실제로 구현된 시스템:
- 플레이어: 이동, 점프, 웅크리기, 사격, 수류탄
- 무기 시스템: 피스톨, HMG, 로켓, 샷건
- 적: 보병, 바주카병, 낙하병 등 여러 종
- 사이드스크롤 카메라
- 이펙트: 스크린셰이크, 히트스톱, 파티클, 점수 팝업

**"동작하는 게임을 만들어라"는 요청에는 AI가 충분히 응답할 수 있음.**

### 2.4 Canvas fallback으로 에셋 없이도 게임이 돌아가게 만들었다

모든 렌더링 함수가 **스프라이트 이미지가 있으면 이미지를, 없으면 Canvas 2D로 직접 그리는** 2중 구조로 되어 있음.
덕분에 에셋 로딩이 실패해도 게임 자체는 동작.

### 2.5 base64 인라인은 단일 파일이 되지만 크기가 비현실적

모든 스프라이트를 base64로 HTML에 인라인하면 설치 없이 바로 플레이 가능하지만 **43MB**.
공유나 로딩 속도 면에서 실용적이지 않음.

---

## 3. 핵심 교훈

1. **에셋 간 디펜던시**(크기, 기준점, 프레임 수)를 맞춰야 자연스러운 애니메이션이 된다 — 단일로 생성하면 안 됨
2. **나무위키 + GitHub 에셋을 인풋으로 제공**하면 원작에 가까운 결과가 나온다
3. **AI가 만든 게임 로직의 완성도는 생각보다는 높다**
